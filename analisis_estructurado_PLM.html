1 Detección de patrones

Recurrencias confirmadas en el código fuente

Cada patrón aparece en al menos 2 módulos independientes. Se descartan las anécdotas de sesión única.

P1 · Inconsistencia de estado entre módulos
🔴 Riesgo alto 4 módulos

El carrito usa localStorage como fuente de verdad pero cada página lo relee de forma independiente. No existe un bus de eventos centralizado ni un singleton observable. Cuando catalogo.js despacha cart:updated, los listeners en otras páginas ya no están activos. Resultado: precios y cantidades que no coinciden entre la página de catálogo y el resumen del carrito.

// Cada módulo lee localStorage por su cuenta — sin singleton
export function getCart() {
  return JSON.parse(localStorage.getItem('plm_cart') ?? '[]');
}

Causa raíz: Ausencia de patrón Pub/Sub o CartStore centralizado. Próximo paso: implementar un CartStore con subscribe() y dispatch().

P2 · Precios como strings vs números en el pipeline
🔴 Riesgo alto Confirmado en producción

Los data-producto-precio del HTML vivían como "2500" (pesos colombianos) mientras formatPrice usaba toLocaleString('es-CO'), y el catálogo mostraba $2.50 (USD). La misma función de formato producía resultados radicalmente distintos según el contexto. Corregido con Intl.NumberFormat('en-US', {currency:'USD'}), pero el patrón raíz persiste: sin validación de tipos al borde del sistema.

Riesgo de regresión: cualquier edición de catalogo.html sin conocer la convención USD vuelve a romper la coherencia de precios.

P3 · CSS que no respeta el sistema de tokens
🟡 Riesgo medio 3 componentes

Los componentes mezclan colores literales con tokens CSS. El caso más crítico: .footer { background: var(--color-text) } — en light mode es carbón oscuro, pero en dark mode ese token se convierte en #f5f0e8 (blanco), produciendo footer blanco con letras blancas. El ícono del carrito tenía opacity: 0.30 hardcodeado. El header ocultaba los botones de tema e idioma con selectores de ID en lugar de clases del design system.

Causa raíz: No existe un lint de CSS que valide el uso de tokens. El design system vive en _variables.css pero no se hace cumplir automáticamente.

P4 · WhatsApp como único canal de conversión sin fallback
🔴 Riesgo alto Estructural

Toda la conversión depende de que el cliente abra WhatsApp correctamente. Si falla la redirección (bloqueador, sin datos móviles, app no instalada), la transacción desaparece sin trazabilidad. El carrito se vaciaba sincrónicamente antes de confirmar que WhatsApp abrió — corregido con setTimeout(clearCart, 1500) pero sigue siendo frágil. Cada fallo de redirección es una venta perdida invisible para el operador.

P5 · Menú mobile con tres implementaciones distintas
🟡 Riesgo medio Historial de 3 sesiones

Panel lateral con transform: translateX → display: flex forzado en .is-open → dropdown con max-height + opacity. La inconsistencia surgió porque el CSS de desktop declaraba display: flex en .header__menu y los breakpoints mobile no lo sobreescribían de forma determinista. Los botones de idioma y tema se ocultaban con selectores de ID (#lang-toggle, #theme-toggle { display: none }) en lugar de clases — una especificidad alta que dificultaba el override.

P6 · Ausencia de validación de esquema en datos de productos
🟡 Riesgo medio Estructural

Los datos de productos viven en atributos HTML sin validación de tipos, rangos ni valores por defecto. Si un data-producto-precio es vacío o no numérico, formatPrice(NaN) produce "$NaN" en el carrito. No existe una función de validación en el borde del sistema que garantice la integridad de los datos antes de añadir al carrito.

2 Necesidades reales

Jobs-to-be-Done: más allá del deseo superficial

El e-commerce no es el canal de compra — es el canal de validación. El cliente usa el sitio para construir confianza antes de comprometerse a ir físicamente.

Cliente — ¿Por qué visita el sitio y no llama directamente?

Deseo superficial: "Quiero ver los productos y hacer un pedido."

  • ¿Por qué el cliente visita el catálogo? → Para ver qué hay disponible hoy.
  • ¿Por qué en el sitio y no llamando? → Porque no quiere interrumpir para preguntar; prefiere decidir solo.
  • ¿Por qué prefiere decidir solo? → Porque siente presión cuando hay un vendedor esperando respuesta.
  • ¿Por qué importa para la conversión? → Si el catálogo no es claro (precios, disponibilidad, foto), el cliente abandona antes de llegar a WhatsApp.
  • ¿Qué necesita realmente? → Certeza visual de que lo que quiere existe y vale lo que espera pagar.

Job real del cliente

"Ayúdame a confirmar que puedo fiar en esta panadería antes de salir de casa."

💡
Implicación de diseño

La página más importante del sitio no es el catálogo general — es la página de producto individual con foto real, precio exacto y señales de confianza (hora de horneado, stock disponible). Esta página no existe aún en el proyecto.

Operador (dueño) — ¿Qué necesita realmente del sistema?

Deseo superficial: "Quiero que los clientes pidan por WhatsApp."

  • ¿Por qué WhatsApp y no llamadas? → Porque queda registro escrito del pedido.
  • ¿Por qué el registro importa? → Porque hornea en cantidad y necesita anticipar cuánto producir.
  • ¿Por qué no puede anticipar ahora? → Porque lee cada mensaje manualmente y no tiene vista agregada.
  • ¿Qué necesita realmente? → Visibilidad de pedidos del día agrupados por horario de retiro.
  • ¿Qué lo bloquea? → No existe interfaz de administración — el sistema actual asume que el operador lee WhatsApp en tiempo real.

Job real del operador

"Dame visibilidad de mis pedidos sin tener que transcribir conversaciones de voz ni leer cada mensaje de WhatsApp por separado."

F1 — Foto de producto deficiente

🟡 Media Las imágenes de productos son fotografías sin tratamiento uniforme: fondo variable, iluminación inconsistente. Font Awesome actúa como fallback pero es un ícono genérico. El usuario no puede evaluar la calidad visual del producto antes de añadirlo al carrito. Frecuencia: cada sesión de catálogo.

F2 — Estado "canasta vacía" confuso

🟡 Media El ícono de canasta tenía opacity: 0.30 — casi invisible. El mensaje "Tu canasta está vacía" no tiene CTA suficientemente prominente. El botón "Ir al catálogo" existe pero es secundario visualmente. Corregido parcialmente: opacity a 0.65.

F3 — Formulario de checkout sin retroalimentación

🟡 Media El formulario tiene campos para nombre, teléfono y horario de retiro, pero no hay indicación de cuántos pasos faltan ni validación en tiempo real visible. El stepper existe en el HTML pero no está implementado funcionalmente. El cliente no sabe si sus datos son correctos hasta el submit.

F4 — WhatsApp como punto único de falla 🔴 Crítica

Si window.location.href falla por cualquier razón (sin datos móviles, app no instalada, browser restrictivo), el pedido se pierde completamente. El cliente ve una pantalla de confirmación con número de orden pero no puede verificar si el mensaje llegó. El operador tampoco recibe señal de que algo falló.

⚠️
Fix implementado — parcialmente efectivo

La pantalla de confirmación con el número de orden LM-YYYYMMDD-XXXX aparece antes de redirigir, lo que permite al cliente anotar la orden y contactar por otro medio. Pero el operador nunca sabrá cuántas redirecciones fallaron.

F5 — Sin confirmación del operador al cliente 🔴 Crítica

El cliente llega a retirar sin saber si el pedido fue aceptado. No hay respuesta automática ni confirmación. El sistema asume que el operador ve el WhatsApp y responde en tiempo real, lo cual no ocurre en horarios de producción temprana (4-6 AM). Esta fricción ocurre después de la conversión pero antes de la retención — un cliente que llega y no encuentra su pedido no vuelve.

🎯
Zona de Oro de Luz Marina

Alta diferenciación de producto + fricción digital moderada-baja + 0% de comisión + relación directa. El checkout de WhatsApp es simultáneamente la limitación técnica del sistema y la ventaja relacional del negocio. El cliente habla directamente con quien hornea — ningún competidor grande puede replicar eso.

Amenaza principal

La amenaza no es Rappi. Es que otra panadería artesanal del mismo sector implemente un sistema similar antes. La ventaja tecnológica actual (trazabilidad, dark mode, checkout sin registro) tiene una vida útil de 6-12 meses antes de que sea replicable con herramientas no-code.

Requerimientos técnicos — Sprint 1

📋 Prompt 1 — Panel Admin

Crear /admin.html con un dashboard que: 1. Solicite contraseña "plm2026" al cargar (sin autenticación real) 2. Lea plm_ordenes del localStorage y muestre tabla ordenada por fecha 3. 📋 Prompt 2 — Stock en vivo

Agregar sistema de stock manual a catalogo.html: 1. Atributo data-producto-stock="disponible|agotado" en cada .producto-card 2. Badge visual sobre imagen: "Disponible" (verde) / "Agotado" (rojo) 3. Deshabilitar botón "Añadir al carrito" si stock === "agotado" 4. En JS: verificar data-producto-stock antes de addToCart() 5. CSS: .producto-card--agotado con overlay semitransparente

📋 Prompt 3 — Validación de esquema

Agregar función validateProductData() en cart.js: 1. Recibe el elemento HTML del producto 2. Valida: precio > 0 y es número, nombre !== '', id !== '' 3. Si falla: throw Error con mensaje descriptivo 4. Envolver addToCart() en try/catch — mostrar error en UI, no alert() 5. Tests: precio NaN, precio negativo, nombre vacío

Síntesis ejecutiva

El proyecto tiene bases técnicas sólidas (modularidad CSS, ES modules, trazabilidad de órdenes) pero tres riesgos estructurales: estado del carrito sin singleton, WhatsApp como punto único de falla, y ausencia de una interfaz para el operador. Las tres oportunidades de Sprint 1 (panel admin, stock en vivo, validación de esquema) son de bajo esfuerzo y alto impacto — ideales para entregar valor en la siguiente sesión de desarrollo.
